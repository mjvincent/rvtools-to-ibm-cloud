# ADR-011: Normalized VM Data Model

## Status
Accepted

## Context
The application now combines RVTools inventory, disk, NIC, memory, pricing,
image readiness, and migration readiness data. Passing loosely structured
dictionaries between parsing, UI, Terraform rendering, and handoff generation
made later feature work riskier because column names doubled as internal API
fields.

The app still needs to preserve existing Streamlit columns, CSV headers,
manifest fields, and Terraform output.

## Decision
Use standard-library dataclasses in `models.py` as the canonical internal VM
data model.

`MigrationVm` is the top-level workload record and includes typed nested records
for source metadata, target recommendation, pricing metadata, image readiness,
memory readiness, migration readiness, disks, NICs, and readiness findings.

Legacy dictionary conversion remains supported through `from_record()` and
`to_record()` so existing UI and output contracts do not change.

## Consequences
New parsing and assessment logic can work with typed objects instead of table
rows. Public output functions can continue accepting legacy dictionaries while
normalizing to `MigrationVm` at their boundaries.

The data model intentionally does not introduce Pydantic or other runtime
dependencies. Validation remains lightweight and compatibility-oriented until a
future schema validation need is explicit.

Old pricing and template experiments are preserved under `experiments/` so
production root files and smoke tests stay focused on supported app behavior.
