# ADR-006: Per-Disk Volume Mapping

## Status
**Accepted / Implemented** (May 2026)

## Context
Earlier Terraform output summarized all RVTools `vDisk` capacity for a VM into a single IBM Cloud block volume. That was useful for cost estimation, but it did not preserve disk role, disk count, controller metadata, or attachment intent. It also left a gap between generated storage volumes and the generated VSI definitions.

Enterprise migrations often need to preserve the distinction between boot disks and data disks. Boot disks are handled through the image import or replication workflow, while data disks should be represented as separately attached block volumes.

## Decision
We now preserve per-disk metadata from RVTools `vDisk` and treat the first discovered disk as the boot/image disk. Additional disks are generated as IBM Cloud block volumes in the storage module and attached to the corresponding VSI in the VSI module.

The storage module exports a per-VM map of data volume IDs. The root module passes that map into the VSI module, where `ibm_is_instance_volume_attachment` resources bind each generated data volume to its target VSI.

The ZIP bundle also includes `disk-mapping.csv` so migration teams can review source disk metadata and target Terraform resource names.

## Consequences
* **Positive**: Multi-disk VMs are represented with separate target data volumes instead of one aggregate volume.
* **Positive**: Generated Terraform now includes explicit data volume attachments.
* **Positive**: Boot disk handling remains aligned to the custom image import workflow.
* **Negative**: Disk ordering is inferred from RVTools disk/unit metadata and should be reviewed during migration planning.
