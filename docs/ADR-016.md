# ADR-016: Terraform Custom Image ID Map for VSI Provisioning

## Status
**Accepted / Implemented** (May 2026)

## Context
The migration handoff package already emits `image-import-variables.tfvars.example` so migration teams can record IBM Cloud VPC custom image IDs after VMware image conversion, Cloud Object Storage staging, and image import. Terraform VSI generation still needed a stable way to consume those IDs without adding pre-import requirements to the Streamlit workflow.

IBM Cloud VPC instances are created from an image ID, and imported custom image IDs are only known after the external image import process completes.

## Decision
Generated Terraform now defines a required `custom_image_ids` map at the root module and passes it into the VSI module. Each generated `ibm_is_instance` reads its image from that map using the same VM key emitted in `image-import-variables.tfvars.example`.

The Streamlit application does not collect custom image IDs before ZIP generation. Users copy the example varfile, replace every placeholder after image import, and pass the populated file to Terraform with `-var-file`.

The `custom_image_ids` variable rejects blank and placeholder values where Terraform validation can evaluate them. Missing VM keys fail during Terraform validation or plan through the module reference.

## Consequences
* **Positive**: The generated Terraform can provision VSIs directly from imported custom images without changing the app build flow.
* **Positive**: The existing handoff file name remains stable for downstream migration teams and automation.
* **Positive**: Image conversion, COS upload, import, and cutover stay external and tool-neutral.
* **Negative**: Terraform planning now requires a populated image ID for every generated VSI.
