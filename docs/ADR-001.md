# ADR 001: Multi-Tab Performance & Resilience Mapping

## Status
**Accepted / Implemented** (April 2026)

## Context
Standard RVTools 'vInfo' snapshots often fail to capture real-time performance bottlenecks. Migrating Virtual Machines (VMs) to IBM Cloud VPC based solely on assigned resources (CPUs/RAM) risks undersizing workloads that are currently experiencing CPU contention or over-provisioning "Zombie" VMs. A single-tab approach ignores the physical constraints of the source cluster, leading to inaccurate Cloud VPC sizing.

## Decision
We implemented a multi-tab correlation engine that pulls from five distinct RVTools sources:
1. **vInfo**: Baseline inventory, power states, and assigned resources.
2. **vCPU**: Real-time performance metrics (Ready %, Co-stop, Overall MHz).
3. **vHost**: Physical host density, core counts, and clock speeds.
4. **vCluster**: Aggregate capacity for N+1 calculation.
5. **vDisk**: Cumulative storage footprint for tiering logic.

### Key Logic Implemented
* **Safety Overrides**: If a VM shows >5% CPU Ready or >3% Co-stop, right-sizing is automatically disabled (forced to 100% allocation) to ensure performance parity in the VPC environment.
* **N+1 Headroom**: Calculation: `(Cluster Total MHz - Largest Host MHz) - Current Demand`. This ensures the destination environment mimics on-premises high-availability standards.
* **Density Guardrails**: Tracking vCPU:pCore ratios to ensure migration profiles stay within the 6:1 density threshold.

## Consequences
* **Positive**: Higher accuracy in VPC profile selection; reduced risk for production migrations; improved business case transparency.
* **Negative**: Requires a full RVTools export (all tabs) rather than a simplified CSV, increasing the data collection requirement.
