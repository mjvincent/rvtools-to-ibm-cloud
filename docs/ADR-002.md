# ADR-002: Modular Infrastructure Output

## Status
**Accepted / Implemented** (May 2026)

## Context
Initial iterations of the tool generated a monolithic `main.tf` file. While functional for simple proof-of-concepts, a single-file approach does not align with IBM Cloud best practices for production environments. As the tool expanded to include complex networking and tiered storage, the code became difficult to audit and hindered "Day 2" operational flexibility.

## Decision
We transitioned the engine to a modular HCL (HashiCorp Configuration Language) output. The tool now generates a structured ZIP bundle containing:
1. **Root Module**: Orchestrates providers and variables.
2. **Networking Module**: Encapsulates VPC, Address Prefixes, and Subnets.
3. **Storage Module**: Manages Block Storage volumes and IOPS tiering logic.
4. **VSI Module**: Manages compute instances and primary network interfaces.

## Consequences
* **Positive**: Aligns with the IBM Cloud Framework for Financial Services; allows for independent updates to networking or storage without re-provisioning compute.
* **Positive**: Supports security group creation and exports subnet/security group mapping values for downstream Terraform instance wiring.
* **Positive**: Adds user override support for profile and storage tier selections, allowing human-directed adjustments to generated Terraform.
* **Positive**: Adds a Terraform override panel to select VPC name, address prefix strategy, and custom subnet CIDRs.
* **Positive**: Adds deployment target awareness for Plain CLI versus IBM Schematics state management.
* **Positive**: Enables the business-case CSV to include mapping-related data with baseline and savings estimates.
* **Positive**: Preserves per-disk `vDisk` metadata and generates separate data disk volumes with VSI attachments.
* **Positive**: Enables direct integration into IBM Cloud Schematics and enterprise CI/CD pipelines.
* **Negative**: Increased complexity in the logic engine to manage cross-module dependency mapping (e.g., passing dynamic Subnet IDs into the VSI module).
