# ADR-009: Memory Readiness and Sizing

## Status
Accepted

## Context
Earlier sizing logic used assigned memory from `vInfo` and CPU utilization to choose IBM Cloud VPC profiles. That is useful for first-pass sizing, but RVTools also provides `vMemory` telemetry that can reveal memory pressure, reservations, limits, and actual active memory.

Memory sizing is risky when source workloads show swapping, ballooning, reservations, or limits. At the same time, workloads with very low active memory can be over-provisioned if the tool only looks at configured memory.

## Decision
Add an advisory memory readiness assessment and use conservative memory guidance during IBM Cloud profile selection.

The app now reads `vMemory` when available and captures:
* Configured memory.
* Active memory.
* Consumed memory.
* Ballooned memory.
* Swapped memory.
* Memory reservation.
* Memory limit.
* Memory hot-add status.

The sizing model preserves configured memory when memory pressure, reservations, or limits are detected. If no pressure exists and active memory is materially below configured memory, the engine can use active memory with a conservative 50 percent configured-memory floor.

## Consequences
Positive:
* IBM Cloud profile recommendations are more defensible because memory telemetry is considered.
* Migration teams get a memory-specific review file, `memory-readiness.csv`.
* Workloads with swapping, ballooning, reservations, or limits are protected from unsafe reductions.

Negative:
* RVTools memory telemetry is point-in-time or recent-observation metadata, not a full performance history.
* Active memory can understate true application needs for bursty workloads.
* Users must still validate profile recommendations with workload owners before Terraform apply.

## Scope
This decision does not add cloud-native monitoring, historical performance ingestion, or automatic memory override approval. It improves advisory sizing and handoff data using RVTools metadata already present in the workbook.
