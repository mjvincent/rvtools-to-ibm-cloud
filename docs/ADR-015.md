# ADR-015: Advisory Network Readiness From Switch and Port Context

## Status
**Accepted / Implemented** (May 2026)

## Context
The application already preserves `vNetwork` rows for multi-NIC mapping and generated IBM Cloud VPC network interfaces. RVTools can also include `vPort`, `dvPort`, `vSwitch`, and `dvSwitch` worksheets with source switch, port group, VLAN/segment, port status, and port-capacity context. Migration teams need that context for review, but it should not silently change target Terraform behavior.

## Decision
We now parse optional `vPort`, `dvPort`, `vSwitch`, and `dvSwitch` worksheets and use them to enrich per-NIC source metadata and advisory network readiness.

`vNetwork` remains the primary source for NIC inventory and Terraform interface generation. Switch and port context adds evidence fields, readiness reasons, handoff columns, and assessment quality coverage only.

Network readiness is advisory. It does not block ZIP generation, alter subnet or security group references, suppress connected NICs, or provision disconnected NICs.

## Consequences
* **Positive**: Migration teams can review source switch type, port group, VLAN/segment, port key, backing source tab, and match confidence before cutover.
* **Positive**: Optional network-detail tabs improve planning confidence without making older RVTools exports unusable.
* **Positive**: Ambiguous or unmatched switch/port rows are retained as review signals instead of guessed.
* **Negative**: Network readiness can flag `Blocked` source evidence while Terraform generation still succeeds, so users must understand that the status is a planning signal rather than an export gate.
