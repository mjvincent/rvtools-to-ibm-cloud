# ADR-007: Multi-NIC Network Mapping

## Status
**Accepted / Implemented** (May 2026)

## Context
Earlier Terraform output mapped each VM to one source network and generated only a `primary_network_interface`. RVTools `vNetwork` provides one row per virtual NIC, including network, switch, adapter type, MAC address, IP address, and connection state. Enterprise workloads may have separate application, database, backup, management, or DMZ interfaces that should remain visible during migration planning.

## Decision
We now preserve `vNetwork` rows as per-VM NIC metadata. The first connected NIC becomes the IBM Cloud VPC primary network interface. Additional connected NICs render as inline secondary `network_interfaces` blocks on the same `ibm_is_instance` resource.

Disconnected NICs are retained in the manifest and `nic-mapping.csv`, but they do not generate active Terraform network interfaces.

## Consequences
* **Positive**: Multi-NIC VMs retain target subnet and security group intent in Terraform.
* **Positive**: Migration teams get a `nic-mapping.csv` review artifact with source IP, MAC, adapter, switch, and connection state.
* **Positive**: Disconnected source NICs are visible without being provisioned accidentally.
* **Negative**: NIC ordering is inferred from RVTools sort, label, and MAC metadata and should be reviewed during migration planning.
